# Code Review Guide

This document defines the checklist that must be followed when reviewing any
code generated by the CrewAI + Aider pipeline. Use it for every PR or commit
produced by an automated sprint.

---

## How to Review a Sprint Output

### Step 1 — Check the git diff

```bash
git diff HEAD~1   # or the specific commit hash
```

Look at every file changed. The diff should match exactly what the sprint
prompt asked for — nothing more, nothing less.

### Step 2 — Run the checklist below for every changed file

### Step 3 — Run the tests

```bash
cd apps/api
npm run test          # unit tests
npm run test:e2e      # e2e tests
```

### Step 4 — Update the feature registry

After approving the PR, update `medicano_crew/docs/specs/feature-registry.md`
with every new file, class, and method added.

## Review Checklist

### Architecture

- [ ] New files follow the `src/<feature>/` folder structure
- [ ] No business logic in controllers — everything delegated to services
- [ ] No raw Mongoose queries in controllers
- [ ] New modules are registered in `AppModule`
- [ ] New modules export their service and `MongooseModule` if needed by others

### Schemas

- [ ] `@Schema({ timestamps: true })` is present
- [ ] Document type is exported: `export type XDocument = X & Document`
- [ ] All ObjectId references use `Types.ObjectId` with a `ref` string
- [ ] `passwordHash` field has `select: false`
- [ ] No plain `string` used for IDs stored in the database

### DTOs

- [ ] All fields have `class-validator` decorators
- [ ] All update DTO fields are marked `@IsOptional()`
- [ ] No `PartialType(CreateDto)` shortcut — update DTOs are explicit
- [ ] No `any` types

### Services

- [ ] Every method that receives an ID calls `Types.ObjectId.isValid(id)` first
- [ ] `NotFoundException` thrown when resource not found
- [ ] `ConflictException` thrown for MongoDB error code `11000`
- [ ] `BadRequestException` thrown for invalid input that passes DTO validation
- [ ] No raw Mongoose errors exposed to the caller
- [ ] No `console.log` — uses NestJS `Logger`

### Controllers

- [ ] All routes protected by `@UseGuards(JwtAuthGuard)` unless explicitly public
- [ ] Public routes are clearly commented as `// public`
- [ ] `@CurrentUser()` used to get userId (not `@Req()`)
- [ ] Return types are explicit — no `any`
- [ ] Logout returns `@HttpCode(204)` with no body

### Guards & Auth

- [ ] JWT payload shape is `{ sub: string; role: Role }`
- [ ] `JwtStrategy.validate()` checks Redis before returning the user
- [ ] No hardcoded secrets — all from `ConfigService`

### Tests

- [ ] Every new service method has at least one unit test
- [ ] Mocks use `jest.fn()` — no real DB or Redis connections in unit tests
- [ ] `afterEach(() => jest.clearAllMocks())` is present
- [ ] E2E tests use `ValidationPipe` with `whitelist: true`

### General

- [ ] No duplicate code — no class, method, or schema recreated that already exists
- [ ] No commented-out code
- [ ] No `any` TypeScript types
- [ ] No `console.log` in production code
- [ ] Imports use correct relative paths (no more than two `../` levels)
- [ ] `feature-registry.md` updated after merge

## Common AI-Generated Issues to Watch For

| Issue | What to look for |
|-------|-----------------|
| Duplicate schema | A new `user.schema.ts` created outside `auth/schemas/` |
| Duplicate enum | A new `Role` enum defined inline instead of imported |
| Duplicate decorator | `@CurrentUser()` recreated instead of imported from `common/decorators/` |
| Missing guard | A route missing `@UseGuards(JwtAuthGuard)` |
| Missing `@IsOptional()` | An update DTO field without `@IsOptional()` |
| Raw error exposed | A `catch` block that re-throws the raw Mongoose error |
| Wrong import path | Importing from `dist/` or using absolute paths |
| Missing module registration | A new module not added to `AppModule` |
| Missing export | A service not exported from its module |
| `onModuleInit` in tests | Tests calling `onModuleInit()` on a service that uses constructor injection |

## Git Commit Format

Every commit produced by the pipeline must follow:

```
<type>(<scope>): <short description>

Types : feat | fix | refactor | test | docs | chore
Scope : NestJS module name (auth, clinics, users, redis, professionals)
```

Examples:
```
feat(auth): add signup and login endpoints
feat(users): add UsersModule with bcrypt hashing
fix(redis): use constructor injection instead of onModuleInit
test(auth): add unit tests for AuthService logout
```

If a commit message does not follow this format, ask the pipeline to regenerate it.
